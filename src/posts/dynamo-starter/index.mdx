## Dynamo 가 만들어진 배경

아마존 닷컴이 전 세계에 걸쳐 서비스를 제공하는 만큼 높은 신뢰성과 확장성을 필요로 했다.
그래서 수백 개의 서비스로 구성된 고도로 분산되고 느슨하게 결합된 서비스 지향 아키텍처를 사용함. 이런 환경에서는 항상 사용가능한
스토리지 기술이 특히 필요함. 예를 들어 고객은 디스크 장애, 네트워크 문제 혹은 데이터 센터 장애에도 불구하고 장바구니에 물건을 추가하거나 쇼핑을 계속 할 수 있어야 한다.
기존 RDB 시스템은 확장성 및 가용성 측면에서 한계가 있었다. 기존 DB 복제 기술은 일반적으로 가용성보다 일관성을 우선시하여 특정 장애 시나리오에서 데이터 사용이 불가한 경향이 있었다.

아마존은 `항상 쓰기 가능`한 데이터 스토어를 필요로 했다. 예를 들어 장바구니 서비스는 서버 장애 중에도 정상적으로 운영되어야 하며, 읽기 시 충돌 해결(conflict resolution of reads) 이라는
복잡성을 감수하는 데이터 스토어가 필요했다.

ACID 보장을 제공하는 전통 데이터 스토어(RDB) 는 가용성이 떨어지는 경향이 있어, Dynamo 는 높은 가용성을 위해 약한 일관성(weak consistency)을 용인하는 애플리케이션을 대상으로 개발됐다.
즉 데이터 자체의 일관성은 떨어질 수 있으나 언제나(99.9%) 가용 가능한 데이터를 제공해야 하는 애플리케이션에서 사용하기 적합하다.
Dynamo 는 특히 매우 엄격한 지연 시간 요구사항을 가지는데, 99.9 백분위 수에서 측정한 성능 목표를 충족해야 했다. 또한 한 번에 하나의 스토리지 호스트(노드) 를 추가하거나 제거할 수 있는 점진적인 확장성을 가져야 했다.
