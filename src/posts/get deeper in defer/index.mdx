# [Golang] Defer 를 알아보자

*defer : 연기하다, 지연시키다


Go 언어의 주요 기능 중 하나인 defer 키워드를 알아보겠습니다.
이 포스트에서는 defer의 개념과 동작 원리를 설명하고, 특히 defer의 스택 기반 실행 순서에 대해 다루겠습니다.



## defer ?

defer 는 Go 언어에서 제공하는 키워드로, 함수의 실행을 지연시키는 데 사용됩니다.
defer 키워드 뒤에 `지연될 함수(deferred function)`를 호출하는 방식으로 사용할 수 있습니다.
만약 `func sample()` 내부에 `defer later()` 라는 defer 구문이 있으면,
sample 함수의 반환/종료 직전까지 later 함수는 호출이 지연됩니다.



주요 특징:

지연 실행: defer로 지정된 함수는 즉시 실행되지 않고, 감싸고 있는 함수가 리턴되기 직전에 실행됩니다.
리소스 관리: 주로 파일, 네트워크 연결 등의 리소스를 안전하게 해제하는 데 사용됩니다.
에러 처리: panic 발생 시에도 defer로 지정된 함수는 실행되므로, 정리 작업을 보장할 수 있습니다.
가독성 향상: 리소스의 획득과 해제를 코드상 가까운 위치에 배치할 수 있어 가독성이 좋아집니다.

```go
func readFile(filename string) error {
    file, err := os.Open(filename)
    if err != nil {
        return err
    }
    defer file.Close()  // readFile() 종료 시 파일을 닫음

    // 파일 읽기 작업...
    return nil
}
```

이 예제에서 defer file.Close()는 readFile 함수가 종료되기 직전에 실행되어 파일을 안전하게 닫습니다.
readFile 함수가 실행되는 동안은 file.Close() 구문을 지연(defer) 하고,
readFile() 이 종료/반환 되기 직전이 돼서야 file.Close() 를 호출합니다.
(함수가 종료될 때 파일을 닫지 않으면 리소스 누수, 열린 파일에 대한 잠금 유지 등 문제가 발생할 수 있습니다.)

`Java` 에서 스트림을 읽거나 파일 리소스를 열면 반드시 `finally` 블럭에서 리소스를 닫아주는 것 처럼
마지막에 Clean-up 작업을 위해 사용된다고 볼 수 있습니다.

## defer 를 써야하는 이유
예시 코드를 하나 더 살펴보겠습니다.
다른 분의 블로그에서 발췌한 코드인데 너무 적절한 예시라 보여드리고 싶었습니다.

##### ☑️ defer 를 사용하지 않을 때
아래 코드는 함수의 모든 return 문이 있는 곳 마다 매번 f.Close() 를 선언해야 합니다.
함수가 반환되는 지점에 일일이 f.Close() 를 작성하는 건 번거러울 뿐더러 누락시 휴먼에러 발생 가능성도 있습니다.

```go
func Contents(filename string) (string, error) {
	f, err := os.Open(filename) // 자원 할당
	if err != nil {
		return "", err
	}
	var result []byte
	buf := make([]byte, 100)
	for {
		n, err := f.Read(buf[0:])
		result = append(result, buf[0:n]...)
		if err != nil {
			if err == io.EOF {
				break
			}
			f.Close() // 지원 해제
			return "", err
		}
	}
	f.Close() // 지원 해제
	return string(result), nil
}



```
[출처: [Go] defer 문의 원리 및 활용](https://bugoverdose.github.io/development/go-defer-basics/)


##### ✅ defer 를 사용할 때
반면 `defer f.Close()` 를 미리 호출하는 경우엔
return 문 마다 일일이 f.Close() 를 선언해둘 필요가 없습니다.
```go
func Contents(filename string) (string, error) {
	f, err := os.Open(filename) // 자원 할당
	if err != nil {
		return "", err
	}
	defer f.Close() // 함수 종료 시점에 자원 해제하도록 '지연 호출 선언'
	var result []byte
	buf := make([]byte, 100)
	for {
		n, err := f.Read(buf[0:])
		result = append(result, buf[0:n]...)
		if err != nil {
			if err == io.EOF {
				break
			}
			return "", err // defer 에 의해 f.Close() 자동 호출
		}
	}
	return string(result), nil // defer 에 의해 f.Close() 자동 호출
}
```


여기까지 defer 의 기본적인 개념을 알아봤습니다.
그럼 만약 하나의 함수 내에 defer 가 여러개라면 어떤 순서로 defer 가 실행될까요?



## defer 대상 함수는 스택 구조로 관리: FILO (First In Last Out)
하나의 함수 내에서 여러개의 defer 함수가 존재하는 경우를 생각해봅시다.

defer 키워드를 만나면 defer 문에 대한 평가가 즉시 일어나고, defer 문에서 지연시키려는 함수를 스택에 추가합니다.

해당 스택에서 지연된 함수들은 선입후출, 즉 스택구조로 관리되다가 한번에 순차적으로 호출됩니다


``` go
func deferDemo() {
    defer fmt.Println("First")
    defer fmt.Println("Second")
    defer fmt.Println("Third")

    fmt.Println("Main function body")
}
// deferDemo() 가 종료되면 "Third" -> "Second" -> "First" 순으로 출력

> Main function body
> Third
> Second
> First
```

주요 포인트:

실행 순서: 여러 개의 defer 문이 있을 경우, 가장 마지막에 defer된 함수가 가장 먼저 실행됩니다.
스택 구조: defer 문을 만날 때마다 해당 함수 호출이 스택에 추가됩니다.
역순 실행: 함수가 종료될 때, 스택의 top부터 순서대로 defer된 함수들이 실행됩니다.






defer 함수의 인자 케이스 탐구
defer를 사용할 때 함수의 인자가 어떻게 평가되고 실행되는지 이해하는 것은 매우 중요합니다. 다양한 시나리오를 통해 defer의 동작을 자세히 살펴보겠습니다.

``` go
func basicDefer() {
	for i := 0; i < 3; i++ {
		defer fmt.Printf("%d \n", i)
	}
	fmt.Println("기본 defer 실행 완료")
}

func deferWithClosure() {
	for i := 0; i < 3; i++ {
		defer func() {
			fmt.Printf("%d \n", i)
		}()
	}
	fmt.Println("클로저를 이용한 defer 실행 완료")
}

func deferWithArgument() {
	for i := 0; i < 3; i++ {
		defer func(n int) {
			fmt.Printf("%d \n", n)
		}(i)
	}
	fmt.Println("인자를 전달하는 defer 실행 완료")
}
```

### for 문에서 defer 를 사용하면 안되는 이유
for 문 내부에서 defer 사용은 일반적으로 권장되지 않으며 IDE 에서도 warning 을 표시합니다.

#### 리소스 누수 위험
defer 는 함수가 종료될 때 까지 실행을 지연시키는데,
for 문 내부에서 defer 사용 시 루프마다 새로운 defer 호출이 스택에 쌓이게 되고 함수가 종료될 때 까지 누적됩니다.
아래 예시 코드에서 루프가 반복될 때 마다 defer file.Close()가 스택에 추가되면 이는 심각한 메모리 문제를 일으킬 수 있습니다.

``` go
func riskyFunction() {
    for i := 0; i < 1000000; i++ {
        file, err := os.Open("some_file.txt")
        if err != nil {
            log.Println(err)
            continue
        }
        defer file.Close() // 위험: 각 반복마다 defer가 쌓임

        // 파일 처리 로직...
    }
}
```

defer 가 누적되면 어떤 메모리 문제가 발생할 수 있을까요? 이를 파악하기 위해선 우선 defer 의 내부 동작을 살펴봐야 합니다.

##### 1. defer의 내부 동작:
Go 런타임은 defer 문을 만날 때마다 defer 레코드라는 작은 데이터 구조를 생성합니다.
이 레코드는 지연된 함수 호출에 대한 정보(함수 포인터, 인자 등)를 포함합니다.
이 레코드들은 함수의 스택 프레임에 연결된 리스트 형태로 관리됩니다.

##### 2. 메모리 사용량 증가:
각 defer 레코드는 작지만, 루프가 수백만 번 반복된다면 메모리 사용량이 급격히 증가할 수 있습니다.
예를 들어, 각 defer 레코드가 32바이트를 사용한다고 가정하면:

1,000,000회 반복 시 약 32MB의 추가 메모리 사용
10,000,000회 반복 시 약 320MB의 추가 메모리 사용

##### 3. 스택 오버플로우 위험:
defer 레코드가 과도하게 쌓이면 스택 오버플로우를 일으킬 수 있습니다.
Go의 기본 고루틴 스택 크기는 2KB에서 시작하여 최대 1GB까지 확장될 수 있지만, 이 한계를 초과하면 프로그램이 중단될 수 있습니다.

##### 4. 성능 영향:
defer 레코드가 많아질수록 함수 종료 시 더 많은 시간이 소요되고 이는 프로그램의 전반적인 성능 저하로 이어질 수 있습니다.

### 해결책과 대안
- 즉시 리소스 해제: 루프 내에서 리소스를 즉시 해제하는 것이 가장 안정.
``` go
func safeFunction() {
    for i := 0; i < 1000000; i++ {
        file, err := os.Open("some_file.txt")
        if err != nil {
            log.Println(err)
            continue
        }
        // 파일 처리 로직...
        file.Close() // 즉시 파일 닫기
    }
}
```

- 함수로 분리 : 루프의 내용을 별도 함수로 분리하고, 그 함수 내에서 defer 를 사용
``` go
func processFile(filename string) error {
    file, err := os.Open(filename)
    if err != nil {
        return err
    }
    defer file.Close() // 이제 안전함

    // 파일 처리 로직...
    return nil
}

func safeLoop() {
    for i := 0; i < 1000000; i++ {
        err := processFile("some_file.txt")
        if err != nil {
            log.Println(err)
        }
    }
}
```

- 고루틴 사용 시 context 통한 관리 :
``` go
func controlledLoop(ctx context.Context) {
    for {
        select {
        case <-ctx.Done():
            return // 컨텍스트 취소 시 루프 종료
        default:
            // 작업 수행...
        }
    }
}
```


### defer 에서 발생한 에러를 처리하라
대부분의 경우 defer 함수에서 리턴된 에러는 무시하지 않는 것이 좋으며,
에러를 직접 처리할지, 호출자에게 전달할지는 문맥에 따라 적절히 판단해야 합니다.
에러를 무실하려면 빈 식별자를 활용하는 것을 추천합니다.
