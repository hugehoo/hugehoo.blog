## 1. GIL이란?

GIL(Global Interpreter Lock)은 CPython 인터프리터 내부 상태를 보호하기 위한 **전역 잠금 장치**다.
전역 잠금 장치? 우선 이 말을 이해하기 전에 1991년 당시 초기 파이썬은 싱글 스레드로 설계되었다는 점을 짚고 넘어가자. 당시 귀도 반 뤼썸은 파이썬을 싱글 스레드 환경을 가정하고 설계했고, 레퍼런스 카운팅을 선택했다. 싱글 스레드 환경은 당연하게도 race condtion 으로부터 자유로웠기 때문에 별도의 락이 필요없었지만, 후에 멀티스레드가 필요해졌을 때 가장 빠르게 안전성을 확보하는 방법으로 GIL 을 도입했다.
CPython 은 프로그램을 실행할 때 컴파일 된 바이트 코드를 스택 기반 인터프리터로 실행하며, 프로세스가 실행되는 동안 일관성 있게 상태를 유지하기 위해 전역 인터프리터 락이라는 방법을 사용해 일관성을 강제로 유지한다.
GIL 이 멀티 스레드 환경에서 안전하게 프로그램을 실행하기 위해 고안된 방법이란 건 이전 내용에서 말했지만, 고민 없이 작성한 멀티 스레드 코드에선 race condtion 이 발생하기 마련이다. 그럼 GIL 은 왜 제 역할을 하지 못하는 걸까? 여기서 중요한건 **GIL 이 무엇을 보호하는가**다.

| 보호 대상                                    | GIL이 보호하는가? |
| -------------------------------------------- | ----------------- |
| CPython 인터프리터 내부 (ob_refcnt 등)       | ✅ 보호함         |
| 개발자가 작성한 공유 변수 (counter, list 등) | ❌ 보호 안 함     |

## 2. GIL은 왜 "파이썬은 느리다"는 오명을 만들었는가?

개발자들 사이에서 흔히 도는 파이썬이 느리다는 오해는 이 GIL 로 부터 시작한다.
물론 인터프리터 언어 특성상 C나 Go 같은 컴파일 언어보다 느린 것은 사실이지만 그 오명의 상당 부분은 GIL로부터 비롯된다.

GIL은 CPython 인터프리터가 한 번에 하나의 스레드만 파이썬 바이트코드를 실행하도록 제어하는 잠금 장치다. CPU 코어가 8개, 16개인 최신 컴퓨터를 쓰고 있더라도, 파이썬 프로세스 안에서는 한 번에 딱 하나의 코어만 일을 할 수 있다. 8차선 도로에 검문소가 하나뿐이라 차가 한 대씩만 지나갈 수 있는 상황과 같다.

<br />

### 멀티코어를 활용할 수 없다는 것

이 제약이 가장 치명적으로 드러나는 건 CPU-bound 작업이다. 데이터 분석, 이미지 처리, 복잡한 수학 연산처럼 CPU를 집중적으로 사용하는 작업에서 일반적인 언어라면 4개의 스레드를 만들어 4개의 코어가 동시에 일하게 할 수 있다. 우리가 흔히 배운 동시성의 개념이다.

하지만 파이썬은 4개의 스레드를 만들어도 GIL 때문에 실제로는 코어 하나가 4개의 스레드를 번갈아 가며 처리한다. 오히려 스레드를 전환하는 비용, 즉 Context Switching 오버헤드 때문에 싱글 스레드보다 더 느려지는 기현상이 발생하기도 한다. 파이썬의 `threading` 모듈을 사용한 멀티스레딩은 실제로 `병렬 처리`가 아닌 아주 빠르게 스레드를 번갈아 가며 실행하는 `동시성` 처리라고 부르는 것이 더 정확하다.

<br />

### 그럼에도 GIL을 유지하는 이유

Go, Java 같은 다른 언어는 GIL 이 없어도 문제없이 동시성 처리가 가능한 것을 보면, 언뜻 성능을 깎는 것처럼 보이는 GIL 을 왜 사용하는지 의문이 들 수 있다. 하지만 이는 초기 파이썬 설계와 관련된 내용이다.

첫째, **메모리 관리의 단순화**다. 파이썬은 Reference Counting으로 메모리를 관리한다. 여러 스레드가 동시에 이 카운트를 수정하면 메모리 오류가 발생할 수 있는데, GIL은 이를 가장 단순하고 안전하게 방어해 준다. 이 부분은 뒤에서 `ob_refcnt`를 다루며 자세히 살펴본다.

둘째, **C 확장 모듈과의 호환성**이다. 파이썬이 초기에 빠르게 성장할 수 있었던 배경에는 C 언어로 작성된 라이브러리들을 쉽게 가져다 쓸 수 있었다는 점이 크다. 그런데 이 라이브러리들 상당수가 GIL의 존재를 전제로 설계되어 있다. GIL을 제거하면 기존의 방대한 생태계가 한꺼번에 깨질 위험이 있었고, 이는 파이썬 커뮤니티가 수십 년간 GIL 제거를 미뤄온 가장 현실적인 이유다.

> CPython 3.13 버전부터 시작해, [GIL 없이 파이썬을 컴파일하는 옵션을 실험 중](https://docs.python.org/3.14/whatsnew/3.14.html#free-threaded-mode-improvements)이다. 이 옵션을 활용하면 GIL 로 인한 제약 없이 파이썬을 실행할 수 있다. 이 기능은 멀티 스레드의 병렬 실행 성능을 향상 시키지만, 많은 수의 C 확장 모듈과 공통 라이브러리들과 아직 호환이 되지 않는 큰 단점이 있다.

<br />

### 오명을 벗기 위한 노력들

파이썬 커뮤니티도 이 한계를 벗어나기 위해 여러 방향에서 극복해 왔다.

가장 대표적인 우회 전략은 **Multiprocessing**이다. 스레드 대신 프로세스를 여러 개 띄워 GIL의 제약을 완전히 피한다. 각 프로세스는 독립적인 GIL을 가지므로 진정한 병렬 실행이 가능하다. 메모리를 더 많이 소비한다는 단점이 있지만, CPU-bound 작업에서는 확실한 선택지다.

웹 크롤링이나 파일 읽기/쓰기 같은 **I/O-bound 작업**에서는 GIL의 영향이 크지 않다. CPU가 대기하는 시간이 대부분이기 때문이다. 이런 경우 `asyncio`를 활용하면 GIL 아래에서도 충분히 효율적으로 동작한다.

그리고 가장 주목할 만한 변화가 있다. Python 3.13부터 도입되기 시작한 **Free-threaded Python** No-GIL 옵션이다. GIL을 선택적으로 비활성화할 수 있는 빌드가 실험적으로 제공되며 수십 년간 이어온 GIL 의 족쇄를 서서히 풀 준비를 하고 있는 셈이다.

## 3. GIL이 있는데도 왜 동시성 이슈가 생기나?

GIL이 멀티스레드 환경에서 인터프리터를 보호한다는 사실을 알고 나면, 자연스럽게 이런 기대를 하게 된다. "그러면 GIL이 알아서 동시성 문제도 막아주는 거 아닌가?" 결론부터 말하면, 아니다.

GIL은 바이트코드를 일정 시간 단위로 스레드 간에 교체하며 실행한다. 문제는 우리가 작성하는 파이썬 코드 한 줄이 실제로는 여러 개의 바이트코드 명령어로 분해된다는 점이다. 예를 들어 `counter += 1`이라는 단순한 한 줄도 내부적으로는 이렇게 동작한다.

```python
LOAD  counter   # 현재 값 읽기
LOAD  1         # 상수 1 로드
ADD             # 더하기
STORE counter   # 결과 저장
```

이 네 단계 사이 어디서든 컨텍스트 스위치가 발생할 수 있다. 두 스레드가 동시에 `counter += 1`을 실행한다고 가정해 보자.

```python
Thread A: counter 읽음 (=0)
                          ← 컨텍스트 스위치
          Thread B: counter 읽음 (=0) → 계산 → counter = 1 저장
Thread A: 계산 → counter = 1 저장  ← 덮어씀!

기대값: 2  /  실제값: 1
```

Thread A가 counter를 읽은 직후 GIL이 Thread B로 넘어가고, Thread B가 같은 값을 읽어 1로 저장한 뒤 다시 Thread A가 돌아와 역시 1로 저장한다. 두 번 증가시켰지만 결과는 1이다. 전형적인 race condition이다.

이처럼 GIL은 **"인터프리터가 죽지 않게"** 하는 장치이지, **"개발자의 공유 변수를 보호해주는"** 장치가 아니다. 공유 자원에 대한 동시 접근을 막으려면 여전히 `threading.Lock()`을 직접 사용해야 한다.

그렇다면 GIL도 결국 Lock의 일종 아닌가? 맞다. 메커니즘 자체는 동일하다. 차이는 **보호 대상의 범위**에 있다. 개발자가 코드에서 사용하는 `threading.Lock()`은 특정 공유 변수를 보호하기 위해 개발자가 직접 지정하는 것이고, GIL은 `ob_refcnt` 같은 인터프리터 내부 값을 보호하기 위해 인터프리터 전역에 고정된 것이다. GIL이라는 이름 자체가 "전역 인터프리터 Lock"인 이유가 바로 여기에 있다.

## 4. GIL이 실제로 보호하는 것 — 레퍼런스 카운팅과 ob_refcnt

GIL이 개발자의 공유 변수를 보호하지 않는다면, 도대체 무엇을 보호하는 걸까? 답은 CPython의 메모리 관리 메커니즘, 구체적으로는 **레퍼런스 카운터(`ob_refcnt`)** 에 있다.

CPython에서 모든 파이썬 객체는 내부적으로 다음과 같은 C 구조체로 표현된다.

```c
typedef struct _object {
    Py_ssize_t ob_refcnt;  // ← GIL이 보호하는 값
    PyTypeObject *ob_type;
} PyObject;
```

`ob_refcnt`는 해당 객체를 현재 몇 군데에서 참조하고 있는지를 나타내는 카운터다. 이 값이 0이 되면 더 이상 아무도 이 객체를 사용하지 않는다는 뜻이므로, CPython은 즉시 메모리에서 해제한다.

이 개념을 직관적으로 이해하려면 도서관을 떠올리면 된다. 파이썬 객체는 도서관의 책이고, `ob_refcnt`는 그 책의 대출자 수다. 대출자가 한 명이라도 있으면 책을 폐기할 수 없고, 대출자가 0명이 되는 순간 폐기 대상이 된다.

코드로 보면 이렇다.

```python
a = [1, 2, 3]   # ob_refcnt = 1 (a가 참조)
b = a            # ob_refcnt = 2 (a와 b 둘 다 같은 객체를 가리킴)
del b            # ob_refcnt = 1
del a            # ob_refcnt = 0 → 메모리 즉시 해제
```

여기서 `b = a`는 리스트를 복사하는 게 아니라는 점이 중요하다. 메모리에 `[1, 2, 3]`은 단 하나만 존재하고, `a`와 `b` 둘 다 그 하나를 가리킨다. `ob_refcnt`가 2인 이유가 바로 이것이다.

```markdown
메모리
┌─────────────┐
│ [1, 2, 3] │ ← ob_refcnt = 2
└─────────────┘
▲ ▲
│ │
a b (둘 다 같은 곳을 가리킴)
```

참고로 `sys.getrefcount(a)`를 호출하면 예상보다 1 큰 값이 반환되는데, 이는 함수 인자로 넘기는 순간 임시 참조가 하나 더 생기기 때문이다. 함수 호출이 끝나면 임시 참조는 즉시 사라지므로 누적되지 않는다.

### GIL이 없다면 어떻게 되는가

이제 핵심 질문에 도달한다. 왜 이 `ob_refcnt`에 GIL이 필요한가?

멀티스레드 환경에서 GIL이 없다고 가정해 보자. 두 스레드가 동시에 같은 객체의 참조를 해제하려 한다.

```markdown
Thread A: ob_refcnt 읽음 (=1)
← 컨텍스트 스위치
Thread B: ob_refcnt 읽음 (=1) → 1-1=0 저장 → 메모리 해제!
Thread A: 1-1=0 저장 → 이미 해제된 메모리를 또 해제 → 크래시
```

Thread A가 `ob_refcnt`를 읽은 직후 Thread B가 끼어들어 같은 값을 읽고, 0으로 만들어 메모리를 해제해 버린다. 그 후 Thread A가 돌아와서 이미 해제된 메모리에 대해 다시 해제를 시도한다. 이는 double free로, 인터프리터 크래시로 이어진다.

GIL은 이 읽기 → 수정 → 저장 과정이 중간에 끊기지 않도록 보호한다. `ob_refcnt`라는 인터프리터 내부의 공유 상태를 안전하게 유지하는 것이 GIL의 본질적인 존재 이유다.

## 5. 왜 Python에만 GIL이 존재하는가?

자연스럽게 드는 의문이 있다. 왜 파이썬에만 이런 개념이 존재하는 걸까? Go에서는 GIL 같은 게 없어도 멀티스레드 프로그램이 잘 동작하지 않는가?

핵심은 **메모리 관리 방식의 차이**다.

CPython은 메모리 해제를 레퍼런스 카운팅으로 수행한다. 객체가 생성되거나 삭제될 때마다 `ob_refcnt`를 읽고 쓰는데, 멀티스레드 환경에서는 여러 스레드가 이 값을 동시에 수정할 위험이 생긴다. 파이썬은 이 문제에 대해 가장 단순한 해결책을 택했다. 인터프리터 전체에 하나의 Lock을 거는 것, 즉 GIL이다.

반면 Go는 GC(Garbage Collector) 방식으로 메모리를 관리한다. 레퍼런스 카운트 같은 공유 카운터가 애초에 존재하지 않는다. GC가 별도의 과정을 통해 주기적으로 "누가 이 메모리를 쓰고 있는가"를 추적하고, 더 이상 참조되지 않는 객체를 정리한다. 스레드들이 공유 카운터를 건드릴 일 자체가 없으니, GIL이라는 개념이 필요하지 않은 것이다.

다른 언어들도 마찬가지다. Java 역시 GC 기반이라 GIL이 없고, Rust는 한 발 더 나아가 컴파일 타임에 소유권 검사를 수행하므로 GC조차 필요 없다. C/C++은 메모리 관리를 개발자에게 맡기는데, 그 대가로 잘못된 접근 시 크래시가 발생할 수 있다.

| 언어   | 메모리 관리             | GIL                     |
| ------ | ----------------------- | ----------------------- |
| Python | 레퍼런스 카운팅         | 있음                    |
| Go     | GC                      | 없음                    |
| Java   | GC                      | 없음                    |
| Rust   | 컴파일 타임 소유권 검사 | 없음 (GC도 없음)        |
| C/C++  | 개발자가 직접 관리      | 없음 (대신 크래시 위험) |

<details>
<summary>Q. 그럼 Go에서는 멀티스레딩을 어떻게 구현하는가?</summary>

파이썬과 달리 Go는 실제로 여러 코어를 동시에 활용할 수 있다. 그 핵심에는 **고루틴(Goroutine)** 이라는 개념이 있다.

Go에서는 스레드라는 용어 대신 고루틴을 사용한다. 가장 큰 차이는 무게다. 파이썬이 사용하는 OS 스레드는 하나당 약 1~2MB의 메모리를 차지하지만, 고루틴은 하나당 단 2KB 정도로 시작한다. 파이썬에서 스레드 수천 개를 만들면 시스템에 심각한 부하가 걸리지만, Go에서는 고루틴 수십만 개를 띄워도 문제없이 동작한다.

더 중요한 건 Go 런타임이 가용한 모든 CPU 코어를 자동으로 활용한다는 점이다. 8코어 CPU라면 물리적으로 8개의 작업을 동시에, 즉 진정한 의미의 병렬(Parallel)로 수행할 수 있다.

Go가 이를 가능하게 하는 비결은 **M:N 스케줄링 모델**에 있다. 다수의 고루틴(G)을 논리적 프로세서(P)가 관리하고, 이를 실제 OS 스레드(M)에 매핑하는 구조다. 특정 코어에서 실행 중인 고루틴이 I/O 대기 등으로 멈추면, Go 스케줄러는 즉시 다른 고루틴을 해당 코어에 할당한다. 여기에 Work Stealing이라는 메커니즘도 더해진다. 놀고 있는 코어가 있으면 바쁜 코어의 작업 큐에서 일을 가져와 처리하므로, 모든 CPU 코어가 유휴 상태 없이 풀가동된다.

코드 레벨에서 보면 그 차이는 더 극명하다.

```python
# Python: threading을 써도 GIL 때문에 한 코어만 바쁨
t1 = threading.Thread(target=task)
t1.start()
```

```go
// Go: 'go' 키워드 하나로 여러 코어에 분산되어 병렬 실행
go task()
```

파이썬은 `threading.Thread`를 만들어도 GIL 아래에서 하나의 코어만 사용하지만, Go는 `go` 키워드 하나로 런타임이 알아서 여러 코어에 작업을 분배한다. GIL이 없기 때문에 가능한 일이다.

</details>

<br />

그렇다면 왜 파이썬은 애초에 레퍼런스 카운팅을 선택했을까? 1991년 파이썬이 만들어질 당시, GC는 구현이 복잡하고 무거운 기술이었다. 레퍼런스 카운팅은 그에 비해 구현이 단순하고, 메모리 해제 시점을 예측할 수 있으며, 다양한 플랫폼에 이식하기 쉽다는 명확한 장점이 있었다. 싱글 스레드 시대에는 완벽한 선택이었다.

그러나 멀티코어 시대가 도래하면서 이 선택은 한계로 돌아왔다. 레퍼런스 카운팅의 동시 수정 문제를 해결하기 위해 GIL이 추가되었고, GIL 때문에 멀티코어를 제대로 활용하지 못하는 구조가 굳어진 것이다. 결국 GIL은 파이썬이 레퍼런스 카운팅이라는 메모리 관리 전략을 선택한 데서 비롯된 필연적인 결과다.

## 6. GIL은 객체의 "존재"를 보호하지, "내용"을 보호하지 않는다

지금까지의 내용을 관통하는 가장 중요한 개념이 하나 있다. GIL이 보호하는 것과 보호하지 않는 것의 경계다.

파이썬에서 `a = [1, 2, 3]`을 실행하면 메모리에는 두 가지가 생긴다. 하나는 `[1, 2, 3]`이라는 데이터 자체이고, 다른 하나는 `ob_refcnt = 1`이라는 카운터다. GIL은 후자만 보호한다. 전자, 즉 데이터의 내용물은 보호하지 않는다.

```python
shared_list = []

# Thread A
shared_list.append(1)   # 리스트 내용 수정 → GIL이 보호하지 않음

# Thread B
del shared_list         # ob_refcnt 감소 → GIL이 보호함
```

GIL 덕분에 `shared_list` 객체가 메모리에서 갑자기 사라지는 일은 없다. 인터프리터가 크래시나는 것은 막아준다. 하지만 그 안의 데이터가 여러 스레드에 의해 동시에 수정되는 것은 전혀 막지 못한다.

> **GIL = "객체가 살아있음"을 보장, "객체의 내용이 안전함"은 보장하지 않음**

한편, 메모리 해제 타이밍에 대해서도 짚고 넘어갈 부분이 있다. 파이썬에서는 `del`을 명시적으로 호출하지 않아도 메모리가 자동으로 해제된다. 함수가 끝나는 순간 지역 변수가 스코프에서 사라지면서 `ob_refcnt`가 0이 되고, 즉시 메모리가 반환된다.

```python
def process():
    data = [1, 2, 3]  # ob_refcnt = 1
    return "done"
# 함수 종료 → data의 ob_refcnt = 0 → 즉시 해제
```

`del`은 함수가 아직 끝나지 않았지만 더 이상 필요 없는 대용량 데이터를 미리 해제하고 싶을 때 사용한다. 10GB 파일을 로드한 뒤 요약 결과만 필요하다면, 원본 데이터를 `del`로 일찍 해제하여 메모리 압박을 줄일 수 있다.

이 즉시 해제 방식은 Go의 GC와 대비된다. Go에서는 함수가 끝나도 메모리 해제는 GC가 별도로 처리하기 때문에 해제 시점을 예측할 수 없다. 파이썬은 `ob_refcnt`가 0이 되는 순간 즉시 해제한다는 점에서 예측 가능성이 높다. 그 대신 이 카운터를 보호하기 위해 GIL이라는 대가를 치르고 있는 것이다.

| 구분   | 해제 시점      | 이유                               |
| ------ | -------------- | ---------------------------------- |
| Python | 함수 종료 즉시 | refcount가 0이 되는 순간 해제      |
| Go     | GC 실행 시점   | refcount 없이 GC가 주기적으로 청소 |

---

## 마무리

정리하면, 파이썬은 레퍼런스 카운팅으로 메모리를 관리하고, `ob_refcnt`는 모든 스레드가 공유하는 값이다. 멀티스레드 환경에서 이 값이 동시에 수정되면 잘못된 시점에 메모리가 해제되어 인터프리터가 크래시난다. 이를 막기 위해 GIL로 한 번에 하나의 스레드만 바이트코드를 실행하도록 강제한다. 그러나 GIL은 인터프리터 내부만 보호할 뿐, 개발자가 작성한 공유 변수는 여전히 직접 Lock으로 보호해야 한다.

GIL은 파이썬의 죄가 아니다. 1991년의 합리적인 선택이 만들어낸 필연적인 트레이드오프이며, 파이썬 커뮤니티는 지금 이 순간에도 그 족쇄를 풀기 위해 노력하고 있다.
