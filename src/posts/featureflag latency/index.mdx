## 1. 99%는 괜찮은데 1%가 문제다
Feature Flag 도입 후 a/b 테스트 등에서 evaluation 결과를 분석할 수 있도록 s3 export 기능을 추가했다. 각 파드에서 1분에 한번씩 축적된 evaluation 결과를 s3 로 보내는 기능으로 평소에는 큰 문제 없었지만 간헐적으로 s3 재시도 문제와 p99 latency 가 튀는 이슈가 있었다. p95 레이턴시는 1ms 내외로 준수한 수준이었는데 p99는 순간 숫자가 약 3초로 뛰어올랐다. 99%의 요청은 깔끔하게 처리되는데 나머지 1%만 3초씩 블로킹되고 있었다.

| 지표 | 값 |
|------|-----|
| API | `GET /피쳐플래그-evaluation` |
| p95 | ~1ms |
| p99 | ~3초 |
| **p95와 p99 차이** | **약 3000배** |

현재 피쳐플래그에서 사용되는 인프라는 feature-flag 메타데이터를 저장하는 redis 와 평가 결과를 저장하는 s3 가 있다. redis 에 저장된 메타데이터는 1분에 한번 씩 조회하여 local cache 에 저장한 뒤 evalutaion 에 사용된다. 다수의 pod 에서 1분에 한번 씩 조회한들 3초 이상의 p99 latency 를 초래하는 요인으로 보긴 어려웠다. 실제로 feature flag 용 레디스를 별도로 운영했기 때문에 latency 지표를 통해 위 상황과는 무관하다는 점을 쉽게 파악할 수 있었다.

## 2. Feature Flag Evaluation API의 구조

원인을 찾기 전에 먼저 이 API가 어떤 경로로 동작하는지 파악할 필요가 있었다. Feature Flag Evaluation API는 클라이언트로부터 특정 flag key에 대한 평가 요청을 받으면 Redis에서 flag 데이터를 조회하여 boolean, string, int 등의 값을 반환하는 단순한 구조다. Redis 조회 자체는 sub-millisecond 수준이므로 평상시에 p95가 1ms 이하인 것도 자연스러운 일이었다.

``` bash
Client ──▶ Feature Flag Service ──▶ Redis (flag 조회)
                  │
                  │ evaluation event
                  ▼
           S3 Exporter (parquet)
```

여기서 눈에 띄는 건 API 응답 경로 외에 존재하는 **DataExporter**다. Feature Flag 라이브러리([go-feature-flag](https://github.com/thomaspoignant/go-feature-flag))는 모든 evaluation 결과를 이벤트로 기록하며, 일정 조건이 충족되면 이 이벤트들을 S3에 parquet 파일로 내보내는 구조를 갖고 있다. 평소에는 이벤트를 메모리 버퍼에 쌓기만 하므로 API 응답에 영향을 주지 않지만, flush가 발생하는 순간에는 상황이 달라진다. 그럼 어떤 상황에서 flush가 발생하고, 그것이 왜 API 레이턴시에 영향을 미치는 걸까?

여기서 의문이 생겼다. Redis가 아니라면 evaluation 함수 내부에서 도대체 뭘 하길래 3초나 블로킹되는 걸까? 코드를 따라가 보니 evaluation 함수는 flag 값을 반환하기 직전에 `DataExporter.AddEvent()`를 호출하고 있었다. 그리고 이 AddEvent()가 특정 조건에서 flush를 트리거하며, flush 동안 mutex를 잡고 동기식으로 S3 업로드까지 수행한다는 사실을 발견했다.

## 4. mutex를 잡고 잠드는 Exporter

go-feature-flag 라이브러리의 DataExporter는 두 가지 조건에서 flush를 트리거한다. 첫 번째는 60초마다 실행되는 타이머 기반 flush이고, 두 번째는 메모리 버퍼에 이벤트가 100,000개 쌓이면 발생하는 버퍼 기반 flush다. 두 경우 모두 flush 과정에서 mutex를 획득한 뒤 동기식으로 S3 Export를 실행하는데, 이 mutex가 문제의 핵심이었다.

| 트리거 | 조건 | 실행 위치 |
|--------|------|----------|
| 타이머 기반 | 60초마다 | daemon goroutine |
| 버퍼 기반 | 100,000개 이벤트 도달 시 | **request goroutine** |

AddEvent()는 모든 evaluation 요청마다 호출되며, 이 함수 역시 이벤트를 버퍼에 추가하기 위해 같은 mutex를 사용한다. flush가 진행되는 동안 mutex가 잠겨 있으므로 그 사이에 들어오는 다른 모든 요청의 AddEvent() 호출이 mutex 앞에서 대기하게 되고, 이것이 곧 API 레이턴시 증가로 이어진다.

``` bash
AddEvent() 호출
     │
     ▼
mutex.Lock()  ◀── flush 중이면 여기서 대기
     │
     ▼
len(cache) >= 100,000?
     │           │
    YES         NO
     │           │
     ▼           ▼
  flush()     cache에
 (동기식)    이벤트 추가
     │
     ▼
S3Exporter.Export()
     │
     ├─▶ applyJitter(): sleep(0 ~ 7000ms)  ◀── 원인!
     │
     └─▶ S3 Upload (~60-70ms)
```

그럼 flush 자체가 느린 건지, 아니면 다른 요인이 있는 건지 확인해볼 필요가 있었다. 디버깅 로그를 추가하여 실제 트래픽을 분석한 결과는 이러했다.

| 지표 | 값 | 분석 |
|------|-----|------|
| event_count | ~5,000 | 버퍼 100,000의 5% 수준 |
| export_interval | ~60초 | 타이머 기반 flush (정상) |
| duration (S3 업로드) | ~60-70ms | S3 업로드 자체는 빠름 |

S3 업로드는 60-70ms 밖에 안 걸렸고 버퍼가 100,000개에 도달한 적도 없었다. 그렇다면 3초라는 시간은 어디서 오는 걸까? 답은 S3 Exporter의 `applyJitter()` 함수에 있었다.

```go
func (f *S3Exporter) applyJitter(ctx context.Context) error {
    if f.MaxJitterMs <= 0 {
        return nil
    }
    jitter := time.Duration(rand.IntN(f.MaxJitterMs)) * time.Millisecond
    select {
    case <-ctx.Done():
        return fmt.Errorf("interrupted")
    case <-time.After(jitter):  // ← 최대 7초 sleep!
        return nil
    }
}
```

Jitter의 원래 의도는 합리적이었다. 여러 pod가 동시에 S3로 업로드하면서 발생할 수 있는 thundering herd 현상을 방지하기 위해 각 pod마다 0~7초의 랜덤 대기 시간을 두는 것이었는데, 문제는 이 sleep이 mutex를 잡은 상태에서 실행된다는 점이었다. flush 동안 mutex를 획득하고 있으니 그 안에서 최대 7초간 잠들면 그 시간 동안 다른 모든 요청의 AddEvent()가 줄 서서 기다리게 된다. 앞서 비유했던 "공용 문을 잠그고 낮잠을 자는 것"이 여기서 정확히 일어나고 있었던 셈이다. mutex라는 문을 잠그고, jitter라는 이름으로 최대 7초간 낮잠을 자는 동안 모든 API 요청이 문 앞에 줄을 서는 것이다.

## 5. 단계적 개선과 지표 변화

원인을 파악한 뒤 여러 PR을 거치며 단계적으로 개선을 진행했다. 모든 PR이 유의미한 레이턴시 개선을 가져온 건 아니었지만, Datadog의 6개월 추이 그래프를 보면 어떤 변경이 실제로 효과가 있었는지 명확하게 확인할 수 있었다.

<br />

### Jitter 추가 — 문제의 시작

첫 번째 PR은 thundering herd를 방지하기 위해 S3 Exporter에 최대 7초의 Jitter를 추가한 것이었다. 의도 자체는 여러 pod가 동시에 flush할 때 S3에 가해지는 부하를 분산시키려는 합리적인 판단이었는데, Export 함수가 동기식으로 실행되고 mutex를 잡고 있다는 점을 간과했다. 이 PR 배포 직후부터 p99 레이턴시가 상승하기 시작했고, 2025년 12월 말에는 일 최대 레이턴시가 **55초**를 넘기며 최악의 상태에 이르렀다.

<br />

### Jitter 비활성화 — 즉각적인 효과

가장 즉각적인 효과를 가져온 건 Jitter를 임시로 비활성화한 PR이었다. 원인이 Jitter라는 확신이 생긴 뒤 Export 함수에서 `applyJitter()` 호���을 주석 처리하는 단순한 변경이었지만 효과는 극적이었다. 배포 직후 일 최대 레이턴시가 55초에서 약 200ms 수준으로 떨어졌고, 이후 지속적으로 안정적인 수치를 유지했다.

```go
func (f *S3Exporter) Export(...) error {
    // temporarily disable jitter to investigate p99 latency causes
    //if err := f.applyJitter(ctx); err != nil {
    //    return err
    //}
    // ...
}
```

| 시기 | 일 최대 레이턴시 | 비고 |
|------|-----------------|------|
| 2025년 11월 초 (Jitter 추가 전) | ~9ms | 정상 |
| 2025년 12월 말 (Jitter 활성 상태) | ~55.8초 | 최악 |
| 2026년 1월 중순 (Jitter 비활성화 후) | ~200ms | 안정화 |
| 2026년 2~4월 (추가 최적화 후) | ~130-180ms | 안정 유지 |

<br />

### FlushInterval 조정

Jitter를 비활성화한 뒤에도 flush가 동기식으로 실행된다는 근본적인 문제는 남아 있었다. 우선 flush 빈도를 줄이기 위해 FlushInterval을 60초에서 3분으로 늘렸다. S3 업로드 자체가 60-70ms 수준이므로 큰 문제는 아니었지만 블로킹 빈도가 줄어드는 것만으로도 tail latency를 안정화시키는 데 도움이 되었다.

```go
return ffclient.DataExporter{
    FlushInterval: 3 * time.Minute,  // 60초 → 3분
    Exporter: &exporter.S3Exporter{
        // ...
    },
}
```

<br />

### AsyncExporter 래퍼 — 비동기 전환

가장 근본적인 서비스 측 개선은 Export 과정을 비동기로 전환한 것이었다. 기존에는 flush가 트리거되면 S3 업로드가 완료될 때까지 mutex를 잡고 기다렸는데, AsyncExporter 래퍼를 도입하여 이벤트 데이터를 복사한 뒤 즉시 mutex를 해제하고 S3 업로드는 별도 goroutine에서 처리하도록 변경했다. 이렇게 하면 flush가 발생하더라도 다른 요청의 AddEvent()가 블로킹되는 시간이 이벤트 복사에 소요되는 마이크로초 수준으로 줄어든다.

## 6. 오픈소스 기여 — mutex hold time 줄이기

서비스 측 개선과 별개로 근본 원인이 go-feature-flag 라이브러리 자체에도 있다는 점이 계속 걸렸다. 라이브러리 내부의 `DataExporter` 스케줄러가 flush 시 mutex를 잡은 채로 Export까지 동기 실행하는 구조 자체가 문제였기 때문이다. 내부 코드를 들여다보니 flush 함수에서 cache를 읽고 Export를 호출하고 다시 cache를 비우는 전체 과정이 하나의 mutex lock 안에서 이루어지고 있었고, 이 구간에서 네트워크 I/O가 발생하니 mutex hold time이 불필요하게 길어지는 것이었다.

이 문제를 해결하기 위해 [go-feature-flag에 PR](https://github.com/thomaspoignant/go-feature-flag/commit/a510291e4c967dce4705d2e2a6ffb3cb80b27900)을 제출했다. 핵심 변경은 mutex hold time을 최소화하는 것으로, cache 데이터를 로컬 변수로 복사한 뒤 즉시 mutex를 해제하고 실제 Export는 mutex 밖에서 실행하도록 수정했다. 이렇게 하면 Export 과정에서 네트워크 I/O가 아무리 오래 걸리더라도 다른 goroutine의 AddEvent() 호출이 블로킹되지 않는다.

이 변경이 라이브러리에 반영되면 Jitter가 활성화된 상태에서도 mutex와 sleep이 분리되어 있으므로 thundering herd 방지라는 원래 목적을 달성하면서도 API 레이턴시에는 영향을 주지 않게 된다.

## 마무리

이번 개선 과정을 돌이켜보면 결국 핵심은 하나였다. **공유 자원을 잠근 채 I/O를 수행하지 마라.** mutex를 잡고 있는 동안 네트워크 호출이나 sleep이 발생하면 그 시간 동안 다른 모든 접근자가 대기하게 되고, 이것이 tail latency로 직결된다. 처음에 비유했던 "공용 문을 잠그고 낮잠을 자는 것"이 정확히 이 패턴이었고, 해결책 역시 단순했다. 문을 잠그는 시간을 최소화하고 잠이 필요하면 문 밖에서 자는 것이다.

p99 같은 tail latency 문제를 조사할 때 기억해둘 만한 점도 있었다. 평균이나 p95로는 보이지 않는 문제가 p99에서 드러나는 경우, 원인은 대부분 "가끔 발생하는 특수 경로"에 있다. 타이머 기반 flush, 버퍼 임계치 도달, 조건부 retry 같은 경로는 평소에는 실행되지 않다가 특정 조건에서만 트리거되므로 평균 지표로는 포착하기 어렵다. Datadog의 flame graph와 레이턴시 추이 그래프를 교차 분석하는 것이 이런 유형의 문제를 찾아내는 데 효과적이었다.
